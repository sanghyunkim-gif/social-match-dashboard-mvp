# KEVIN Dashboard UI Design Audit

> 작성일: 2026-03-14
> 대상: Social Match Dashboard MVP (KEVIN)
> 분석자: UI Designer Agent (CDO Division)

## 1. 헤더 영역 (Header)

### 구조
- 좌측: 브랜드명 "KEVIN" (h1, Sora 폰트 36px) + 서브 텍스트 "지표 중심 분석을 위한 스마트 대시보드"
- 우측: "데이터 소스: Supabase" 라벨 + 사용자명 + 로그아웃 버튼
- `position: sticky; top: 0; z-index: 120` 로 스크롤 시 고정

### 잘 된 점
- sticky 헤더로 항상 브랜드와 사용자 상태가 보여 맥락 유지 가능
- 로그아웃 버튼이 우측 상단에 위치해 일반적인 패턴과 일치
- Sora 디스플레이 폰트로 브랜드명에 시각적 차별화 부여

### 문제점

| # | 이슈 | Severity | Category |
|---|------|----------|----------|
| H-1 | **"데이터 소스: Supabase" 라벨이 내부 구현 디테일 노출** -- 운영팀 사용자에게 의미 없는 기술 정보. 화면 공간 낭비. | minor | navigation |
| H-2 | **sticky 헤더 배경이 `linear-gradient`로 body 배경과 동일하게 설정됨** -- 스크롤 시 콘텐츠가 헤더 아래로 비칠 수 있음 (반투명 영역). 실제로 gradient 위에 gradient가 겹치면서 시각적 구분이 약해짐. | major | layout |
| H-3 | **헤더와 컨트롤바 모두 sticky인데 두 영역의 시각적 경계가 불명확** -- 둘 다 같은 배경색, 구분선 없음. 사용자가 어디까지가 "고정" 영역인지 인지하기 어려움. | major | layout |
| H-4 | **모바일(720px 이하)에서 헤더가 `flex-direction: column`으로 변경되나 sticky 높이 계산이 동적이 아닐 수 있음** -- ResizeObserver로 보완했지만 전환 시점에 깜빡임 가능. | minor | layout |

## 2. 컨트롤바 (ControlBar)

### 구조
- 3개 행(row)으로 구성:
  1. **템플릿 행**: 템플릿 선택 드롭다운 + 저장 버튼
  2. **메인 필터 행**: 기간단위(disabled) + 기간범위 + 측정단위(Segmented) + 필터 + 초기화 + 조회 버튼
  3. **지표 행**: 지표 선택 버튼 + 활성 지표 칩 목록 + 전체 해제

### 잘 된 점
- 템플릿 저장/불러오기/수정/삭제 기능이 컨트롤바 내에 통합되어 워크플로우가 자연스러움
- 측정단위를 Segmented Button으로 표현하여 선택지가 한눈에 보임
- 선택된 지표가 칩으로 표시되어 현재 상태를 즉시 확인 가능
- 초기화 버튼 제공으로 원점 복귀가 쉬움

### 문제점

| # | 이슈 | Severity | Category |
|---|------|----------|----------|
| C-1 | **기간단위 드롭다운이 "주"로 고정되어 있으나 disabled 상태로 노출** -- 사용자가 왜 비활성인지 혼란. disabled UI가 불필요하게 공간을 차지하며, 향후 확장 전까지는 숨기거나 읽기전용 텍스트로 대체하는 것이 나음. | minor | navigation |
| C-2 | **"조회 및 AI 자동 분석" 버튼명이 길고 역할이 모호** -- 단일 버튼에 두 가지 행위(조회 + AI 분석)를 함축. 사용자가 "AI 분석만 하고 싶은데?"라고 생각할 수 있음. | minor | navigation |
| C-3 | **3개 행의 시각적 계층이 약함** -- 템플릿 행, 메인 필터 행, 지표 행이 같은 카드 안에 있으나 구분선만으로 분리. 정보 우선순위가 불명확함. "어디서부터 설정해야 하는가?"라는 사용 순서 안내가 부재. | major | layout |
| C-4 | **필터 드롭다운(`select`)이 측정단위="전체"일 때 옵션이 "전체" 하나뿐** -- 의미 없는 드롭다운이 노출. 측정단위가 "전체"일 때 필터 드롭다운을 숨기거나 비활성화하는 것이 적절. | minor | navigation |
| C-5 | **측정단위 Segmented에 `region_group`, `region`, `court` 옵션이 types.ts에 정의되어 있으나 UI에서는 미노출** -- 향후 추가 가능하나, type과 UI 간 불일치가 존재. | minor | navigation |
| C-6 | **모바일에서 측정단위 세그먼트 버튼들이 `flex-wrap`으로 줄바꿈되면서 수직 공간 과다 사용** -- 5개 버튼이 2~3줄로 감싸지며 컨트롤바 전체 높이가 크게 증가. sticky 영역이 화면의 40% 이상을 차지할 수 있음. | major | layout |
| C-7 | **초기화 버튼이 `btn-ghost` 스타일로 시각적 존재감이 약함** -- 기능적으로 중요하지만 "조회" 버튼 바로 옆에 배치되어 실수로 누를 위험. 위치 또는 시각적 구분이 필요. | minor | layout |

## 3. 결과 테이블 (MetricTable / EntityMetricTable)

### 구조
- **MetricTable**: 측정단위="전체"일 때 사용. 행=지표, 열=주차. Sparkline + 히트맵 배경색.
- **EntityMetricTable**: 특정 측정단위 선택 시 사용. 행=엔티티x지표, 열=주차.
- CSS Grid 기반 레이아웃 (`data-grid` + `data-row` + `data-cell`). 컬럼 리사이즈 지원.
- 수평 스크롤 (`table-scroll`). 주차 수에 따라 최소 너비 720px.

### 잘 된 점
- Sparkline + 히트맵 배경색으로 추이를 시각적으로 빠르게 파악 가능
- 컬럼 리사이즈 기능으로 사용자가 데이터 밀도를 조절 가능
- delta 값에 색상 구분(파란=상승, 빨간=하락)으로 변화 방향이 명확
- formatValue로 number/percent 형식이 자동 적용

### 문제점

| # | 이슈 | Severity | Category |
|---|------|----------|----------|
| T-1 | **테이블 헤더가 스크롤 시 고정되지 않음** -- `data-header`에 `position: static`이 명시적으로 설정됨. 주차가 많은 경우(24주) 수평 스크롤 시 헤더 주차 라벨이 사라져 어떤 주차 데이터를 보고 있는지 알 수 없음. | critical | navigation |
| T-2 | **EntityMetricTable에서 엔티티가 많을 때 수직으로 매우 길어짐** -- 엔티티 x 지표 조합으로 행이 급증. 접기/펼치기(accordion) 또는 엔티티 탭/필터 부재. | major | layout |
| T-3 | **EntityMetricTable에서 같은 엔티티의 연속 행에서 엔티티명이 `visibility: hidden`** -- 시각적 그룹 구분이 약함. 엔티티 간 경계선이나 배경색 교차가 없어서 어떤 행이 어떤 엔티티에 속하는지 빠르게 인지하기 어려움. | major | layout |
| T-4 | **컬럼 리사이즈 핸들이 시각적으로 거의 보이지 않음** -- `1px` 너비의 회색 선. 사용자가 리사이즈 가능하다는 사실을 발견하기 어려움(discoverability 부족). | minor | navigation |
| T-5 | **히트맵 배경색이 단일 색상(파란 계열)만 사용** -- 퍼센트 지표와 숫자 지표가 같은 색 계열로 표현. 지표 유형별 시각적 구분이 없음. | minor | color |
| T-6 | **data-cell의 font-size가 12px, 11px으로 매우 작음** -- 긴 시간 분석 작업에 눈 피로를 유발. 특히 delta 값(11px)은 가독성이 떨어짐. | major | typography |

## 4. AI 채팅 위젯 (AiChat)

### 구조
- 우하단 고정 아바타 버튼 (56x56, floating, 호흡 애니메이션)
- 클릭 시 채팅 패널 슬라이드업 (400px width, max 70vh)
- 헤더: 아바타 + "KEVIN AI" + "Assistant" 뱃지 + 닫기
- 메시지 영역: 버블 UI (user=파란, assistant=회색)
- 추천 질문 칩 (초기 3개 + 동적 추천)
- 입력: 텍스트 + 전송 버튼
- 필터 자동 적용 + 분석 요청 자동화

### 잘 된 점
- 대시보드 컨텍스트를 자동으로 AI에 전달하여 별도 설명 불필요
- 추천 질문 칩이 초기 사용 진입 장벽을 낮춤
- 필터 자동 적용 후 자동 분석 요청 흐름이 매끄러움
- ESC 키로 닫기 가능

### 문제점

| # | 이슈 | Severity | Category |
|---|------|----------|----------|
| A-1 | **아바타 버튼이 에러 로그 토글 버튼과 겹침** -- 아바타는 `bottom: 84px`, 에러 로그는 `bottom: 24px`. 근접해 있어 실수 클릭 가능. 특히 모바일에서 아바타가 `bottom: 72px`으로 더 가까워짐. | major | layout |
| A-2 | **채팅 패널이 에러 로그 패널과 z-index 충돌 가능** -- 에러 로그 z-index: 200, 아바타: 250, 채팅 패널: 260. 에러 로그 패널이 열려 있을 때 채팅 패널이 그 위를 덮지만, 에러 로그 토글 버튼은 채팅 아바타 아래에 위치하여 동시 사용이 혼란스러움. | major | layout |
| A-3 | **채팅 히스토리가 세션 중에만 유지** -- 페이지 새로고침 시 모든 대화 소실. 내부 운영팀이 이전 분석 결과를 다시 참조하고 싶을 때 불편. | minor | navigation |
| A-4 | **아바타 호흡 애니메이션(`ai-breathe`)과 부유 애니메이션(`ai-float`)이 동시에 적용** -- 두 애니메이션 모두 transform을 사용하여 서로 덮어쓸 수 있음(CSS specificity 경쟁). `has-pulse` 클래스 적용 시 breathe 제거되지만 일관성이 부족. | minor | layout |
| A-5 | **모바일에서 채팅 패널이 `left: 8px; right: 8px`으로 전체 너비를 차지** -- 대시보드 조작이 불가능. 패널 닫기 전에는 대시보드 데이터를 볼 수 없어 분석 대화 중 데이터 참조가 어려움. | major | layout |

## 5. 에러 로그 패널 (ErrorLogPanel)

### 구조
- 우하단 고정. 토글 버튼 (둥근 검정 pill, "Error Log (N)")
- 클릭 시 320px 패널 표시 (어두운 배경, 최대 320px 높이)
- 각 로그: message + time + detail(pre)

### 잘 된 점
- 에러를 비침습적으로 수집하여 디버깅에 유용
- 최대 50건 제한으로 메모리 관리
- 비우기 기능 제공

### 문제점

| # | 이슈 | Severity | Category |
|---|------|----------|----------|
| E-1 | **운영팀 사용자에게 Error Log가 항상 노출** -- 개발/디버깅 용도의 UI가 프로덕션에서도 보임. 운영팀이 "Error Log (3)"을 보면 불안감을 느낄 수 있음(실제로는 console.error 캡처일 뿐이어도). | major | navigation |
| E-2 | **에러 로그 토글 버튼이 AI 아바타와 같은 우하단에 위치** -- 두 고정 요소가 수직으로 겹치며 공간 경쟁. | major | layout |

## 6. 지표 선택 모달 (Metric Picker)

### 구조
- 오버레이(어두운 반투명) + 우측 슬라이드 패널 (max 560px, 전체 높이)
- 헤더: "지표 선택" + 닫기
- 본문: 지표 카드 목록 (이름 + ID + 설명 + 쿼리 복사 버튼)
- 푸터: 선택 초기화 + 선택완료

### 잘 된 점
- 사이드 패널 패턴으로 대시보드 컨텍스트를 유지하면서 지표 탐색 가능
- 각 지표에 설명과 ID가 표시되어 정보 접근성이 좋음
- 쿼리 복사 기능이 개발/분석 워크플로우에 유용
- 오버레이 클릭으로 닫기 가능

### 문제점

| # | 이슈 | Severity | Category |
|---|------|----------|----------|
| M-1 | **지표 수가 많을 때 검색/필터 기능이 없음** -- 스크롤만으로 원하는 지표를 찾아야 함. 현재 6개 정도이면 괜찮지만, 지표가 10개 이상으로 늘어나면 사용성 저하. | minor | navigation |
| M-2 | **선택된 지표 수 카운터가 없음** -- 몇 개를 선택했는지 확인하려면 스크롤하며 파란 테두리를 확인해야 함. 푸터에 "N개 선택됨" 표시가 필요. | minor | navigation |
| M-3 | **모달 z-index(320)가 충분히 높지만 fetch-overlay(400)보다 낮음** -- 데이터 로딩 중 모달이 열려 있으면 fetch-overlay에 가려질 수 있음. 동시 사용 시나리오에서 혼란 가능. | minor | layout |

## 7. 로딩/페치 오버레이

### 구조
- 전체화면 고정 오버레이 (`inset: 0, z-index: 400`)
- 중앙 카드: 스피너 + "데이터를 불러오는 중입니다..." + 실행취소 버튼

### 잘 된 점
- 실행취소 버튼으로 사용자가 긴 요청을 중단 가능
- AbortController 연동으로 실제 네트워크 요청 취소

### 문제점

| # | 이슈 | Severity | Category |
|---|------|----------|----------|
| L-1 | **전체화면 블로킹 오버레이가 과도함** -- 데이터 조회 중 앱 전체가 잠김. 필터 변경이나 다른 탐색이 불가. 인라인 로딩(테이블 영역만 스켈레톤)이 더 적절. | major | navigation |
| L-2 | **실행취소 버튼이 너무 작고 눈에 띄지 않음** -- 12px 폰트, 작은 pill 형태. 오버레이가 떠 있을 때 유일한 탈출구인데 시각적 존재감 부족. | minor | navigation |

## 개선 계획

### P0 (반드시 수정) -- 사용성 차단 이슈

| # | 이슈 | 권장 조치 |
|---|------|-----------|
| T-1 | 테이블 헤더 미고정 | `data-header`에 `position: sticky; top: var(--table-sticky-top)` 적용. CSS에 이미 `--table-sticky-top` 변수가 계산되어 있으므로 `position: static` 제거만으로 해결 가능. |
| L-1 | 전체화면 블로킹 오버레이 | fetch-overlay를 테이블 영역 내부 인라인 로딩(스켈레톤 또는 반투명 오버레이)으로 교체. 컨트롤바 조작은 가능하도록 유지. |

### P1 (수정 권장) -- 사용성 저하 이슈

| # | 이슈 | 권장 조치 |
|---|------|-----------|
| H-2, H-3 | sticky 헤더 배경 + 구분선 부재 | sticky 영역에 `background: var(--bg)` 단색 또는 `backdrop-filter: blur(8px)` 적용. 하단에 1px 구분선 또는 미세 shadow 추가. |
| C-3 | 컨트롤바 행 계층 약함 | 3개 행에 시각적 단계를 부여 -- 템플릿 행은 secondary 색상, 메인 행은 primary 강조, 지표 행은 neutral. 또는 각 행에 라벨 아이콘 추가. |

## UX 권장사항

### 1. 정보 계층 구조 (Information Hierarchy)

현재 페이지는 단일 스크롤 뷰에 모든 기능이 직렬 배치되어 있습니다:

```
헤더 → 컨트롤바(템플릿 → 필터 → 지표) → 결과 테이블 → [AI 채팅]
```

**권장 구조:**

```
헤더 (sticky)
├── 컨트롤바 (sticky, 단순화)
│   ├── 핵심 필터 (기간 + 측정단위 + 필터) -- 항상 보임
│   └── 확장 영역 (지표 선택 + 템플릿) -- 토글로 접기 가능
├── 결과 요약 (KPI 카드 3~4개) -- 조회 후 테이블 상단에
├── 결과 테이블
└── AI 채팅 (플로팅)
```

- **핵심 필터와 부가 필터를 분리**: 자주 변경하는 기간/측정단위는 항상 노출, 지표 선택과 템플릿은 접기 가능하도록.
- **KPI 요약 카드 추가**: 테이블 진입 전에 핵심 수치(진행률, 전주 대비 변동 등)를 카드 형태로 보여주면 사용자가 "무엇이 중요한가"를 빠르게 파악.

### 2. 네비게이션 구조 제안

현재 단일 페이지 앱이므로 별도 라우팅은 불필요하지만, **컨트롤바 내부의 네비게이션 흐름**을 개선할 수 있습니다:

- **단계적 안내**: 첫 사용 시 "1. 기간 선택 → 2. 측정단위 선택 → 3. 조회" 순서를 시각적으로 안내 (번호 뱃지 또는 진행 표시).
- **최근 조회 이력**: 컨트롤바에 "최근 조회" 드롭다운 추가. 사용자가 이전 필터 조합으로 빠르게 돌아갈 수 있도록.

### 3. 인터랙션 패턴 제안

| 현재 | 개선안 | 이유 |
|------|--------|------|
| 전체화면 로딩 오버레이 | 인라인 스켈레톤 + 컨트롤바 활성 유지 | 사용자가 로딩 중에도 필터를 수정하여 다음 조회를 준비할 수 있음 |
| 조회 버튼 클릭 필수 | 필터 변경 시 자동 조회 (debounced) 또는 "변경사항 있음" 뱃지 | 수동 조회 패턴은 유지하되, 변경 후 조회를 잊어버리는 상황 방지 |
| 에러 메시지가 카드 형태로 결과 위에 표시 | Toast/Snackbar 패턴으로 3~5초 후 자동 사라짐 | 에러가 결과 위에 고정되면 성공적 재조회 후에도 이전 에러가 남아있어 혼란 |
| AI 채팅 닫기 = 분석 맥락 단절 | 채팅 패널 최소화(접기) 상태 추가 | "마지막 AI 응답 요약"을 아바타 옆 말풍선으로 표시하여 맥락 유지 |

## 종합 평가

### 강점
- **기능 완성도가 높음**: 필터 → 조회 → 테이블 → AI 분석 → 템플릿 저장의 전체 워크플로우가 구현됨.
- **CSS 변수 기반 디자인 시스템**: 색상, 폰트, 그림자, 반경 등이 변수로 관리되어 일관성 유지 용이.
- **컴포넌트 분리가 적절**: ControlBar, MetricTable, EntityMetricTable, AiChat, ErrorLogPanel 등 역할별 분리.
- **접근성 기본 요소 갖춤**: `aria-label`, `aria-pressed`, `role`, `tabIndex` 등 기초적 접근성 속성 사용.

### 핵심 개선 영역
1. **테이블 헤더 고정 (P0)** -- 가장 시급. 데이터 분석 작업에서 헤더 없이 스크롤하는 것은 사용성을 심각하게 저해.
2. **로딩 UX (P0)** -- 전체 블로킹 대신 인라인 로딩으로 전환하여 사용자 통제감 유지.
3. **고정 요소 정리 (P1)** -- 우하단에 AI 아바타 + 에러 로그가 겹치는 문제. 역할 분리 필요.
4. **모바일 대응 (P1)** -- sticky 영역 과다, 채팅 패널 전체 점유 등 소형 화면 경험 개선.
5. **정보 계층 재정비 (P1)** -- 컨트롤바 3행의 우선순위 명확화, KPI 요약 카드 추가.
